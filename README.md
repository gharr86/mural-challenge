# Mural challenge

## Stack

- **Runtime / language:** [Node.js](https://nodejs.org/), [TypeScript](https://www.typescriptlang.org/)
- **UI:** [React 18](https://react.dev/), [styled-components](https://styled-components.com/)
- **Build & dev server:** [Vite 5](https://vitejs.dev/)
- **Data fetching:** [TanStack Query](https://tanstack.com/query) — REST client reads notes from [json-server](https://github.com/typicode/json-server) using `fetch`
- **Utilities:** [date-fns](https://date-fns.org/), [nanoid](https://github.com/ai/nanoid)
- **Mock API:** json-server serves `db.json` on port **3000**
- **Unit tests:** [Jest](https://jestjs.io/), [React Testing Library](https://testing-library.com/react)
- **E2E tests:** [Cypress](https://www.cypress.io/)
- **Linting / formatting:** ESLint (Airbnb + TypeScript + Prettier), Prettier

## How to run

1. **Install Node.js**  
   Use a current LTS version (this repo’s `json-server` dependency may log an engine warning below its stated minimum; use **Node 22.12+** if you want a clean install with no warnings).

2. **Install dependencies** (from the project root):

   ```bash
   npm install
   ```

3. **Start the app and the mock API** in one terminal:

   ```bash
   npm run dev:full
   ```

   This runs Vite on **http://localhost:5173** and json-server on **http://localhost:3000** with `db.json`.

   Alternatively, use two terminals:

   ```bash
   npm run dev
   ```

   ```bash
   npm run mock:server
   ```

4. Open **http://localhost:5173** in the browser.


## Tests

### Unit tests (Jest)

```bash
npm test
```

Watch mode:

```bash
npm run test:watch
```

### E2E tests (Cypress)

Cypress uses **http://localhost:5173** as `baseUrl`. The UI must load notes from json-server, so **start the stack first** (for example `npm run dev:full`).

Interactive runner (pick browser and specs):

```bash
npm run cypress:open
```

Headless run (CI-style):

```bash
npm run cypress:run
```

The first time you run Cypress on a machine, it may download the Cypress binary; wait until that finishes.

## How you approached the challenge and scoped the work

I read carefuly through the job deliverables and ordered tasks based on functionality, tried to set the ground first and then mount things. From base project scaffolding and stack options to specific structure details, I tried to always have a working app throughout the process.

## What assumptions you made

I tried to assume as little as possible. My goal was to meet the required features as sharp as possible without over complicating myself. I found the assignment very clear.

## Frontend architecture: component structure, state management, and key technical decisions

I decided to implement atomic design for components, since this type of app is likely to have few routes/screens/pages. Since the data structure is rather small I opted for using react query instead of redux, also for the same reason as the atomic design choice (not necessary to access data from a complex component tree).
For component creation, I decided to go as light as possible and used styled components. I thought of bringing a lib like MUI mostly for accessibility features, but thinking this should be performant as it grows, I decided to code a bit more when creating components but keeping bundle small and being able to highly customize things.
I unit tested all the util functions that contain pure logic. For component functionality I set up an e2e suite using cypress.

## Performance and accessibility: what you implemented and what you'd do next

My goal was to make everything suitable for keyboard navigation and compatible with screen readers. Other features like adequate the app palette for compliant color contrast ratios would be a good next step

## UX decisions

I visualized a simple layout. Just a canvas where the notes are rendered and a top bar with the filters. Future controls and other improvements would be located in the top bar as well.

## If you used AI, how you used it

Most of the code was generated by AI (I used cursor). I try to split large tasks into concise, assertive prompts, my previous planning stage before starting the code part had this implementation as primary goal.
After each code generation, I went through everything and adjusted what I considered necessary.
